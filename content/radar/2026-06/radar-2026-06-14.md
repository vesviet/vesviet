Chào mừng trở lại với bản tin **Tech Radar**. Trong kiến trúc Microservices hiện đại, việc duy trì một hệ thống có thể giao tiếp linh hoạt cả ra ngoài (HTTP) lẫn nội bộ (gRPC) là yêu cầu thiết yếu. Đồng thời, việc quản lý trạng thái (State Management) trong môi trường phân tán đòi hỏi các giải pháp chống đụng độ dữ liệu cực kỳ khắt khe.

Hôm nay, chúng ta sẽ "mổ xẻ" cách kết hợp framework **Kratos** đình đám của Go và **Dapr v1.15** để giải quyết trọn vẹn bài toán này.

---

## 1. Kratos Dual-Protocol: HTTP & gRPC chạy song song

**Answer-first:** Kratos framework tích hợp với Dapr v1.15 State Management thông qua sidecar pattern, cho phép chạy song song HTTP và gRPC server. Để tránh đụng độ trạng thái (state collisions) khi chạy dual-protocol, hệ thống sử dụng Dapr ETags thông qua `SaveStateWithETag` để kiểm soát đồng thời lạc quan (Optimistic Concurrency) và dùng Middleware để đồng bộ Metadata.

### Protocol-First Design với Protobuf
Kratos sử dụng Protobuf làm "nguồn chân lý" duy nhất (single source of truth). Từ một file `.proto`, Kratos tự động generate code cho cả gRPC và HTTP RESTful API. Điều này giúp các lập trình viên không phải viết logic định tuyến thủ công cho từng loại giao thức.

### Unified Transport Layer (Giải quyết bài toán DRY cho Middleware)
Điểm sáng kiến trúc của Kratos nằm ở lớp Transport trừu tượng. Thay vì phải cấu hình Middleware (như Logging, Authentication, Tracing) riêng biệt cho HTTP và gRPC, bạn chỉ cần viết một lần. Lớp Unified Transport sẽ áp dụng đồng nhất các Middleware này cho cả 2 server, loại bỏ hoàn toàn mã lặp (DRY).

## 2. Dapr v1.15 State Management: Pluggable Architecture & ETags

**Answer-first:** Dapr v1.15 quản lý trạng thái qua kiến trúc pluggable, giúp trừu tượng hóa database bên dưới (Redis, PostgreSQL). Đặc biệt, Dapr cung cấp cơ chế Optimistic Concurrency Control (OCC) sử dụng ETags để ngăn chặn triệt để tình trạng ghi đè dữ liệu (lost updates) trong môi trường phân tán.

### Sức mạnh của Dapr Sidecar
Dapr chạy dưới dạng sidecar tách biệt hoàn toàn khỏi logic ứng dụng. Khi Kratos cần lưu trữ trạng thái, nó không cần quan tâm database bên dưới là MongoDB hay Redis. Kratos chỉ đơn giản gọi API của Dapr qua HTTP hoặc gRPC trên `localhost`.

### Cơ chế Optimistic Concurrency Control (OCC) với ETags
Trong mô hình Dual-Protocol của Kratos, một request từ HTTP và một từ gRPC có thể cùng lúc chỉnh sửa một record. Để giải quyết, Dapr gắn một `ETag` vào mỗi bản ghi trạng thái. Khi Kratos cập nhật dữ liệu, nó phải gửi kèm ETag hiện tại. Dapr sẽ từ chối giao dịch nếu ETag không khớp (nghĩa là dữ liệu đã bị sửa đổi bởi tiến trình khác).

## 3. Tích hợp Kratos và Dapr: Giải quyết State Collisions & Metadata

**Answer-first:** Để tích hợp hoàn hảo, lập trình viên sử dụng Dapr Go SDK gọi hàm `SaveStateWithETag` kết hợp cùng Kratos Middleware để truyền tải `context.Context` xuyên suốt. Điều này đảm bảo mọi tracking ID và ETag được giữ nguyên khi di chuyển giữa các lớp HTTP/gRPC và Dapr Sidecar.

### Code snippet: `SaveStateWithETag` bằng Dapr Go SDK
Khi lưu dữ liệu, bạn cần bắt buộc lấy ETag trước, xử lý logic, và lưu lại kèm theo ETag đó.

```go
item, err := client.GetState(ctx, storeName, key, nil)
// ... xử lý logic tăng counter ...
err = client.SaveStateWithETag(ctx, storeName, key, data, item.Etag, nil)
```

### Lưu ý sống còn về Metadata Propagation
Kratos chuyển đổi thông tin từ HTTP headers và gRPC metadata vào `context.Context`. Để Dapr Sidecar nhận diện được các thông tin tracing hay auth token này, bạn **bắt buộc** phải truyền `ctx` này vào mọi hàm gọi Dapr SDK. Việc bỏ quên context sẽ làm đứt gãy luồng Distributed Tracing.

## 4. Q&A: Những vướng mắc thực tế (FAQ)

**Answer-first:** Tích hợp Kratos và Dapr mang lại sự linh hoạt nhưng đòi hỏi phải tối ưu cấu hình mạng local, quản lý vòng đời ứng dụng (graceful shutdown) và xây dựng chiến lược retry hợp lý để giảm thiểu độ trễ (latency overhead).

{{< faq q="Dapr Sidecar có làm tăng latency của Kratos gRPC không?" >}}
Có, nhưng rất nhỏ (khoảng ~2.1ms ở p99). Để tối ưu cho các hệ thống tải cao, bạn nên cấu hình Kratos giao tiếp với sidecar qua kênh gRPC (`app-channel`) và bật tính năng `keep-alive` để tái sử dụng connection.
{{< /faq >}}

{{< faq q="Test Kratos và Dapr local bằng Docker Compose thế nào?" >}}
Bí quyết là đặt container của Dapr sidecar sử dụng chung network interface với container Kratos bằng khai báo `network_mode: "service:kratos-app"`. Điều này giả lập chính xác mô hình mạng của Pod trên Kubernetes.
{{< /faq >}}

{{< faq q="Cấu hình Graceful Shutdown cho Kratos và Dapr thế nào?" >}}
Việc đóng ứng dụng cần được phối hợp chặt chẽ. Đảm bảo tham số `terminationGracePeriodSeconds` trên Kubernetes phải lớn hơn giá trị annotation `dapr.io/graceful-shutdown-seconds` của Dapr để tránh việc sidecar bị kill trước khi Kratos xử lý xong request cuối cùng.
{{< /faq >}}

{{< faq q="Dapr State Management có hỗ trợ Query/Filter dữ liệu không?" >}}
Có, Dapr cung cấp một API `/query` (hiện ở giai đoạn alpha). Nó hỗ trợ các toán tử `EQ`, `IN`, `AND`, `OR` trên dữ liệu JSON. Tuy nhiên, tính năng này chủ yếu tương thích với các database hỗ trợ query JSON phức tạp như MongoDB hay PostgreSQL.
{{< /faq >}}

{{< faq q="Làm sao để auto-retry khi cập nhật State gặp lỗi 409 Conflict?" >}}
Thay vì tự viết các vòng lặp exponential backoff rườm rà bên trong Kratos, bạn nên tận dụng tính năng Resiliency của Dapr. Khai báo các chính sách retry trực tiếp trong file `resiliency.yaml`, Dapr sẽ tự động xử lý khi ETag bị lỗi 409 Conflict.
{{< /faq >}}
